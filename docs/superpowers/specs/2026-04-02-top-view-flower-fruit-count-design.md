# 俯视角花朵与果实计数设计

## 一、背景

当前项目会基于草莓单株的三个 RGB 视角图像进行表型分析，已经输出的指标包括：

- 叶面积
- 绿度
- 凸包面积
- 冠层高度
- 植株冠径
- 侧视投影面积

本次新增需求已经确认如下：

- 增加花朵和果实识别
- 将识别结果接入现有分析流程
- 在“表型测量结果”中显示
- 在导出结果中一并输出

本次第一版的范围已经明确：

- 只使用 `TOP` 俯视图
- 只输出 `flower_count` 和 `fruit_count`
- 不做花朵阶段分类
- 不做青果、红果等多类别果实分类

## 二、目标

- 新增两个基于俯视图的表型指标：`flower_count`、`fruit_count`
- 复用现有的色卡校正结果和 TOP 植株分割结果
- 花果识别逻辑与现有植株分割逻辑解耦
- 生成可用于调参和排查误检的调试图
- 保持与现有 GUI 结果表、批量导出、单样本导出兼容

## 三、不在本次范围内的内容

- 不引入深度学习检测模型
- 不做多视角融合去重
- 不识别花蕾
- 不识别青果
- 不导出单个花或单个果实的逐目标明细
- 不做 GUI 大改版

第一版只要求把花朵数和果实数稳定接入现有结果链路。

## 四、用户可见结果定义

### 1. 花朵数

`flower_count` 定义为：俯视图中可见的开放白花数量。

第一版的判定规则定义为：

- 候选区域颜色接近白色，且亮度较高
- 候选区域必须位于已分割出的植株冠层掩膜内部
- 候选区域通过面积和形状过滤
- 相互粘连的花朵允许通过分裂算法拆分后再计数

该指标只统计俯视图中“看得见”的白花。被叶片完全遮挡的花不在本次范围内。

### 2. 果实数

`fruit_cout` 定义为：俯视图中可见的成熟红果数量。
n
第一版的判定规则定义为：

- 候选区域颜色为红色或橙红色
- 候选区域必须位于已分割出的植株冠层掩膜内部
- 候选区域通过面积和形状过滤
- 相互粘连的果实允许通过分裂算法拆分后再计数

该指标第一版只面向“可见成熟红果”。青果不在本次范围内。

## 五、整体架构

### 1. 集成位置

新逻辑应放在现有植株分割逻辑旁边，而不是直接塞进当前的冠层分割函数里。

建议的模块边界：

- 新增模块：`core/organs.py`
- `core/segmentation.py` 继续只负责植株冠层分割
- `core/traits.py` 继续负责将算法结果转换为表型指标
- `core/pipeline.py` 负责串联新的花果识别步骤，并把结果写入 `TraitResult`

### 2. 建议的数据结构

在 `core/organs.py` 中新增：

- `OrganInstance`
  - `label_id: int`
  - `area_pixels: int`
  - `centroid_xy: tuple[float, float]`
  - `bounding_box: tuple[int, int, int, int] | None`

- `TopFlowerDetectionResult`
  - `status: str`
  - `message: str`
  - `count: int`
  - `mask: np.ndarray`
  - `labeled_mask: np.ndarray`
  - `instances: list[OrganInstance]`
  - `overlay_image: np.ndarray`
  - `debug_images: dict[str, np.ndarray]`

- `TopFruitDetectionResult`
  - 结构与 `TopFlowerDetectionResult` 一致

同时在 `PlantAnalysisResult` 上增加：

- `top_flower_detection: Any | None = None`
- `top_fruit_detection: Any | None = None`

这样后续调试图、结果导出、GUI 展示都能直接访问对应结果。

## 六、数据流

整体流程如下：

1. 读取三视角图像
2. 执行现有色卡校正
3. 执行现有 TOP 植株冠层分割
4. 如果 TOP 分割成功，则在 TOP 图像上执行花朵识别
5. 在同一张 TOP 图像上执行果实识别
6. 将计数结果映射为新的表型指标
7. 保存花果识别调试图
8. 在 GUI 和导出结果中显示这两个新指标

花果识别使用的图像优先级如下：

- 优先使用校正后的 TOP 图像
- 如果色卡校正失败，则退回原始 TOP 图像
- 无论哪种情况，都必须受 TOP 植株掩膜约束

## 七、识别算法设计

### 1. 共用预处理

花朵识别和果实识别都应复用以下输入和步骤：

- 使用校正后的 TOP 图像
- 使用 TOP 植株掩膜作为搜索范围限制
- 在颜色阈值提取前增加一次轻量高斯去噪
- 使用连通域分析作为最终目标计数基础

两类识别都应输出以下调试阶段图：

- 冠层区域限制后的源图
- 原始颜色候选掩膜
- 清理后的掩膜
- 分裂辅助图，例如距离变换图或 marker 图
- 最终实例标签图
- 最终叠加可视化图

### 2. 花朵识别

花朵识别优先寻找“高亮、低饱和、接近白色”的区域。

建议规则链如下：

- 将 TOP 图像转换到 HSV 和 Lab 空间
- 组合条件生成白花候选掩膜：
  - 亮度高
  - 饱和度低
  - 偏黄或偏绿程度受限
- 与植株冠层掩膜取交集
- 删除过小连通域
- 执行一次闭运算，连接碎裂花瓣区域
- 在最终计数前执行基于距离变换的分裂步骤
- 对保留下来的实例做计数

该方案的优点：

- 适合当前项目这种受控拍摄条件下的可见白花计数

该方案的局限：

- 高光反射可能被误判为花
- 如果冠层掩膜过于宽松，浅色背景干扰可能漏进来
- 花朵严重重叠时仍可能少计

### 3. 果实识别

果实识别优先寻找“红色、橙红色”的成熟果区域。

建议规则链如下：

- 将 TOP 图像转换到 HSV 和 Lab 空间
- 使用红色双区间色相范围，加上最小饱和度和亮度阈值，生成候选掩膜
- 与植株冠层掩膜取交集
- 删除过小连通域
- 根据面积和填充率过滤细碎红色伪目标
- 在最终计数前执行基于距离变换的分裂步骤
- 对保留下来的实例做计数

该方案的优点：

- 适合成熟红果的可见计数

该方案的局限：

- 红色标签、夹子或其他人工物可能被误识别为果实
- 被叶片部分遮挡时容易粘连或漏计
- 青果第一版不计入结果

## 八、表型指标映射

需要在 `core/pipeline.py` 的 `TRAIT_SPECS` 中新增：

- `TraitSpec("flower_count", "花朵数", (VIEW_TOP,), "count")`
- `TraitSpec("fruit_count", "果实数", (VIEW_TOP,), "count")`

状态规则如下：

- 检测成功时，无论结果是否为 0，都标记为 `computed`
- 如果 TOP 冠层分割失败，则两个指标都不计算
- 如果色卡校正失败但仍基于原始 TOP 图运行，则仍标记为 `computed`，但消息中必须说明结果可信度较低

结果消息要求：

- 明确写出当前使用的是校正后图像还是原始图像
- 明确写出这是“基于俯视图可见目标”的计数，不代表整株总花果数

## 九、对现有模块的修改要求

### 1. Pipeline

需要在 `core/pipeline.py` 中完成：

- 在 TOP 分割成功后执行花朵识别
- 在 TOP 分割成功后执行果实识别
- 将识别结果保存到 `PlantAnalysisResult`
- 把 `flower_count`、`fruit_count` 写入 `TraitResult`
- 在最终汇总消息中加入花朵数和果实数

### 2. GUI

第一版不要求新增专门的花果预览卡片。

预期效果：

- “表型测量结果”表格总行数增加 2 行
- 用户可以在表中看到 `花朵数` 和 `果实数`

### 3. 导出

`utils/exporter.py` 当前是基于 `result.trait_map()` 动态导出，因此主要要求是：

- 确保新指标已经进入 `TRAIT_SPECS`
- 确保中文表头能正确映射
- 确保 CSV 和 Excel 中都包含 `flower_count` 与 `fruit_count`

## 十、调试图设计

新增两个调试输出分组：

- `flower_count`
- `fruit_count`

每组至少包含：

- 冠层限制后的源图
- 原始候选掩膜
- 清理后掩膜
- 分裂辅助图
- 实例标签图
- 最终叠加图

第一版采用的是阈值法，这批调试图是必要产物，不是可选项。

## 十一、异常与失败处理

- 如果 TOP 回输入缺失，则花朵数和果实数保持未计算
- 如果 TOP 分割失败，则花朵数和果实数保持未计算
- 如果识别流程执行成功但未找到目标，则输出 `0`
- 如果色卡校正失败，则退到原始 TOP 图像继续计算，并在消息中提示可信度较低
- 如果花果识别模块自身抛异常，则只影响花果这两个新指标，不应让已有冠层类表型全部失败

原则是：尽量保留部分成功结果，不因为新增模块影响已有主流程结果。

## 十二、测试方案

### 1. 器官识别单元测试

为新的 `core/organs.py` 增加测试，至少覆盖：

- 冠层掩膜内部存在白花样目标时，能正确计数
- 冠层掩膜内部存在红果样目标时，能正确计数
- 小噪声不会误报
- 两个相连目标可通过分裂步骤拆成两个
- 冠层掩膜外部目标不会被计入

这些测试优先使用合成图像，保证稳定、可复现、执行快。

### 2. Pipeline 测试

扩展现有 pipeline 测试，验证：

- `flower_count` 和 `fruit_count` 会出现在结果列表中
- TOP 花果识别成功时，结果值能正确写入
- TOP 分割失败时，只让这两个指标保持未计算，不影响已有指标

### 3. 导出测试

扩展导出测试，验证：

- CSV 中包含 `flower_count` 和 `fruit_count`
- 调试导出模式下，新指标的状态和消息字段存在
- Excel 表头包含“花朵数”“果实数”

## 十三、风险与权衡

- 阈值法对拍摄条件和颜色一致性敏感
- 白花与高光反射在视觉上接近，容易误检
- 红果计数只代表俯视图中可见成熟果，不等于整株总果数
- 如果冠层掩膜切得太狠，边缘花果可能会被漏掉

这些限制在第一版是可接受的，因为当前明确目标就是“轻量、俯视角、只做计数”。

## 十四、实现边界

这次实现只覆盖：

- 一个新的花果识别模块
- 两个新的表型指标
- pipeline 接入
- 导出接入
- 对应自动化测试

不包括：

- GUI 大改
- 模型训练
- 多视角融合
- 花蕾识别
- 青果识别
- 更细粒度器官分类

## 十五、验收标准

- 结果表中出现 `花朵数` 和 `果实数`
- 单样本和批量导出结果中都包含这两个字段
- 算法仅基于 TOP 俯视图运行
- 花朵数统计可见开放白花
- 果实数统计可见成熟红果
- 未检测到目标时导出值为 `0`，而不是空值
- 现有冠层类表型结果不被破坏
- 自动化测试覆盖新的花果识别与导出行为
