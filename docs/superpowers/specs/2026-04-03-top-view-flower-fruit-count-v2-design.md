# 俯视角花朵与果实计数 V2 设计

## 一、为什么要做 V2

当前第一版 `flower_count / fruit_count` 主要基于颜色候选区和连通域计数。

这条路线已经暴露出两个关键缺陷：

- 叶片、茎秆、反光高亮区域容易被误判为花朵
- 同一朵花的白色区域被裂成两个连通域后，容易被重复计数

因此第二版不再把“颜色候选区个数”直接当作最终计数，而是改成：

- `颜色` 只负责生成候选
- `结构` 和 `轮廓` 负责确认是不是一个真实器官实例
- `实例中心` 负责最终计数

## 二、参考文献带来的设计结论

本次方案并不是直接照抄某一篇论文，而是结合文献中的共同方向，做成适合当前项目的非深度学习版本。

文献里比较一致的趋势如下：

- 草莓花检测的经典路线也不是纯颜色阈值，而是 `超像素 / 区域合并 / 区域级分类`
- 草莓花检测和果实检测的主流已经逐步转向 `目标检测` 或 `实例分割`
- 对花果这种容易遮挡、重叠、反光的目标，领域里更重视 `轮廓结构`、`实例中心` 和 `邻域上下文`

对本项目的直接启发是：

- 不能再把“白色连通域数”定义为花朵数
- 不能再把“红色连通域数”定义为果实数
- 第一版虽然不引入训练模型，但必须把 `结构约束` 和 `实例分裂 / 合并` 加进去

## 三、V2 的总体原则

### 1. 花朵

花朵数不再由白色花瓣区域直接决定，而改为：

`花朵数 = 满足结构约束的花朵实例数`

一个花朵实例应尽量满足以下特征组合：

- 存在一个黄色花心候选
- 花心周围存在白色花瓣候选
- 白色区域与黄色中心在空间上形成邻近关系
- 整体轮廓大小和几何特征落在合理范围内

换句话说，第二版的计数锚点是“花心驱动的实例”，不是“白块驱动的连通域”。

### 2. 果实

果实数改为：

`果实数 = 满足红果结构约束的果实实例数`

果实实例应尽量满足：

- 颜色为红色或橙红色
- 轮廓封闭且实体感较强
- 面积、短轴、填充率、圆整度、凸度落在合理范围内
- 与细长叶柄、碎片、标签、边缘噪声区分开

### 3. 输入范围

第二版仍然保持当前产品边界：

- 只使用 `TOP` 俯视图
- 只统计 `可见开放花` 与 `可见成熟红果`
- 仍然受现有 `TOP 冠层掩膜` 约束
- 仍然优先使用颜色校正后的图像

## 四、花朵数 V2 设计

## 4.1 目标定义

`flower_count` 定义为：俯视图中可见开放花的实例数。

实例级定义改为：

- 一朵花是否成立，优先看是否存在 `黄色花心 + 相邻白色花瓣`
- 白色高光没有花心支撑时，不计入花朵
- 一个花心对应多个白瓣碎片时，应视作同一朵花
- 一个大白块中若存在两个可分离花心，应尝试拆成两朵

## 4.2 处理流程

### 第一步：在冠层掩膜内提取黄色花心候选

在 HSV / Lab / BGR 空间联合提取黄色区域：

- 色相落在黄到黄绿范围
- 饱和度较高
- 亮度中高
- 排除明显偏暗、偏褐、偏土壤色区域

之后做：

- 开运算去噪
- 小区域过滤
- 连通域标记

输出：

- `yellow_center_mask`
- `yellow_center_labels`
- `yellow_center_instances`

这一步是花朵实例的核心锚点。

### 第二步：提取白色花瓣候选

白色候选仍然使用现有的“高亮、低饱和、中性白”方向，但只作为支持证据。

规则建议：

- HSV 饱和度低
- V / Lab-L 高
- RGB 三通道差值较小
- Lab 的 a/b 偏移不能太大

之后做：

- 闭运算连接碎裂花瓣
- 小区域过滤
- 细长区域过滤

输出：

- `white_petal_mask_raw`
- `white_petal_mask_cleaned`

### 第三步：基于花心邻域组装花朵实例

对每一个黄色花心候选，构建其邻域搜索区：

- 以花心质心为中心做圆形或椭圆形扩张
- 扩张尺度与花心面积、图像分辨率相关

在该邻域内寻找相邻白色花瓣区域。

若满足以下条件，则认为构成一个花朵实例：

- 花心附近存在足够面积的白色花瓣
- 白色区域与花心的最短距离在阈值内
- 白色区域不是极细长叶脉反光

这样做可以直接排除：

- 叶片表面的纯白高光
- 茎秆上的亮斑
- 与花心无关的独立白噪声

### 第四步：花朵实例合并

解决“同一朵花被裂成两个白块”的问题。

合并规则：

- 若多个白色区域都归属于同一个黄色花心，则合并为一个花朵实例
- 若一个黄色花心的邻域里出现多个小白块，但这些白块彼此接近，并围绕同一个中心分布，仍然只计作一朵

这里的计数单位必须是：

- `一个花心主导的一朵花`

而不是：

- `几个白色区域 = 几朵花`

### 第五步：多花粘连分裂

解决“两朵花贴在一起”的问题。

当一个白色大区域中存在多个黄色花心时：

- 先以黄色花心作为 marker
- 再在白色花瓣区域上做 marker-controlled watershed
- 将粘连大区域拆成多个花实例

如果白色区域粘连，但只有一个可靠黄心，则优先视作一朵花，不盲目分裂。

## 4.3 花朵结构特征

每个候选花朵建议计算以下结构特征，用于过滤误检：

- 花心面积
- 花瓣总面积
- 花心到花瓣的平均距离
- 花瓣包围角覆盖率
- 外接矩形短边
- 圆整度
- 凸度或 solidity

建议优先保留以下目标：

- 有黄心
- 有白瓣
- 轮廓不是细长条
- 花瓣分布不是单侧极端偏置

建议优先删除以下目标：

- 无黄心的高光白斑
- 极细长反光条
- 过小孤立白噪声
- 颜色虽白但周围没有花器结构支持的区域

## 五、果实数 V2 设计

## 5.1 目标定义

`fruit_count` 定义为：俯视图中可见成熟红果的实例数。

第二版仍然不做青果统计，但要从“红色块计数”升级为“红果实例计数”。

## 5.2 处理流程

### 第一步：颜色候选

先提取红色 / 橙红色候选区：

- HSV 红色双区间
- 最小饱和度和亮度约束
- 可叠加 Lab-a 通道偏红约束

输出：

- `fruit_color_mask_raw`

### 第二步：轮廓级过滤

对每个候选连通域计算：

- 面积
- 外接矩形短边
- 长宽比
- 填充率
- 圆整度
- convex hull ratio / solidity

过滤掉：

- 细长碎片
- 反光边缘
- 细长叶柄
- 小红噪声
- 非实体的空心轮廓

### 第三步：粘连分裂

对于较大的红果候选区：

- 距离变换
- 局部峰值检测
- marker-controlled watershed

目的是把粘连红果拆成多个果实例，而不是把一整团成熟果当作一个目标。

### 第四步：上下文约束

为了减少误检，增加上下文规则：

- 候选必须位于冠层掩膜内
- 与色卡、标签、花盆边缘相邻的异常红区优先剔除
- 过于靠边且不满足实体形状的目标优先剔除

## 六、建议的数据结构扩展

现有 `core/organs.py` 已经有：

- `OrganInstance`
- `TopFlowerDetectionResult`
- `TopFruitDetectionResult`

第二版建议新增或扩展的字段：

- `OrganInstance`
  - `confidence_score: float | None = None`
  - `shape_metrics: dict[str, float] | None = None`
  - `support_label_ids: tuple[int, ...] = ()`

其中：

- 花朵的 `support_label_ids` 可记录花心和花瓣的关联 label
- 果实的 `shape_metrics` 可记录圆整度、solidity、aspect_ratio 等

如果不想扩展公开 dataclass，也至少要在 `debug_images` 或中间结构里保留这些信息，方便调参与回归测试。

## 七、调试图要求

第二版必须强化调试图，不然无法调参数。

### 花朵调试图

至少保存：

- `canopy_source`
- `yellow_center_mask`
- `white_petal_mask_raw`
- `white_petal_mask_cleaned`
- `flower_seed_overlay`
- `flower_merge_result`
- `flower_split_markers`
- `flower_labeled_mask`
- `flower_overlay`

### 果实调试图

至少保存：

- `canopy_source`
- `fruit_color_mask_raw`
- `fruit_mask_cleaned`
- `fruit_shape_filtered`
- `fruit_split_markers`
- `fruit_labeled_mask`
- `fruit_overlay`

## 八、与现有第一版实现的主要差异

相较于当前版本，第二版的核心变化是：

### 花朵

- 不再直接对白色候选区做连通域计数
- 改为先找黄色花心，再结合白色花瓣做实例组装
- 增加“同花合并”和“多花分裂”

### 果实

- 保留颜色候选，但增加更强的轮廓结构约束
- 更明确地区分“红果实体”和“红色碎片 / 细长假目标”

## 九、实现优先级

建议按以下顺序落地：

1. 花朵黄心提取
2. 花朵黄心-白瓣组装
3. 同花合并与多花分裂
4. 果实轮廓过滤增强
5. 花果调试图补全
6. 回归测试集补全

## 十、回归测试要求

第二版必须新增针对真实失败模式的回归测试。

### 花朵

- 叶片高光不计花
- 茎秆高光不计花
- 一朵花裂成两个白块仍计 1
- 两朵贴近花在两个黄心存在时能计 2

### 果实

- 细长红条不计果
- 小碎红块不计果
- 单个成熟果能计 1
- 两个粘连成熟果能拆成 2

## 十一、长期路线

从文献趋势看，真正更稳的路线仍然是：

- 花朵：实例检测或实例分割
- 果实：实例分割或轮廓学习

因此第二版的定位应当明确：

- 它是“从纯颜色计数升级到结构化规则计数”的过渡版
- 不是最终形态

第二版的价值在于：

- 在不引入训练数据的前提下，显著降低当前这类明显误报
- 给后续如果要引入模型，提前理顺实例级数据流和调试体系

本规格中的 `黄心锚点 + 白瓣结构 + 轮廓过滤` 属于基于这些方向推导出的、适合当前项目约束的工程化规则方案。
